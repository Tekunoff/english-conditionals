# Gerund / Infinitive / Bare — Fast Prod Blank UX Design

**Date:** 2026-03-20  
**Status:** Approved (fast MVP for production review)

## Goal

Ускоренно вывести на прод новый UX:
- на экране задания показывать английское предложение с пропуском (`____`);
- ниже давать выбор формы глагола (`doing` / `to do` / `do`);
- для кейсов с разным смыслом (`both-forms`) добавить отдельный выбор перевода.

## Scope (Fast MVP)

В первой итерации для быстрого согласования на проде:
- обновить `gerund-only.html`;
- обновить `both-forms.html`;
- обновить `mix.html`;
- не трогать в этой итерации `infinitive-only.html` и `bare-infinitive.html`.

## UX Flow

### Базовый поток (обычные группы)
1. Открытие карточки группы.
2. Показ английского предложения с пропуском: `I finished ____ files`.
3. Ниже три кнопки формы: `doing`, `to do`, `do`.
4. Поведение:
   - неверный выбор: выбранная красная, правильная зелёная;
   - верный выбор: зелёная подсветка + кнопка `Далее`.
5. Переход к следующему вопросу.

### Поток для `both-forms`
1. Шаг формы (как выше).
2. После верного выбора формы — дополнительный шаг выбора перевода:
   - вопрос «Выбери правильный перевод»;
   - 2 варианта: правильный и альтернативный смысл.
3. После верного перевода — `Далее`.

## Data Design

Текущая структура данных сохраняется.

Используем существующие поля:
- `words`;
- `verbForm`;
- `ru`;
- `note`;
- `keyVerb`.

### Производное поле в рантайме
- `enTemplate` вычисляется на клиенте из `words` и `keyVerb`.
- Пример: `I finished organizing files` → `I finished ____ files`.

### Дополнительно для `both-forms`
- добавить `translationOptions: [correctRu, altRu]`.
- Используется только во втором мини-шаге.

## Error Handling

- Пустой датасет/битая запись: показать fallback `No tasks available`, не падать.
- Если не удалось вычислить `enTemplate` (нет `keyVerb`):
  - безопасный fallback: скрыть пропускной режим для этой записи и показать обычную строку;
  - запись логируется в консоль как warning.
- При неверном ответе прогресс не двигается.

## Progress & Storage

- Ключи localStorage остаются прежними (`gerundOnly`, `bothForms`, `verbFormsMix` и т.д.).
- Счётчик «изучено» обновляется только после полного успешного прохождения вопроса:
  - обычные группы: после верного выбора формы;
  - `both-forms`: после верного выбора формы + верного выбора перевода.

## Test Strategy

### Smoke (обязательно)
- `index` открывается;
- `gerund-only` показывает английскую строку с `____` и 3 формы;
- `both-forms` после выбора формы показывает блок выбора перевода;
- `mix` работает в формате `____` + выбор формы.

### E2E (обновить существующие)
- `exercise.spec.js`: проверка blank-режима вместо chip-сборки.
- `validation.spec.js`: неверная/верная форма.
- `mix.spec.js`: шаг формы в blank-режиме.
- `note.spec.js`: для `both-forms` проверять переводный мини-шаг.

## Acceptance Criteria

- На проде в `gerund-only` видно английское предложение с пропуском и выбор из 3 форм.
- На проде в `both-forms` после выбора формы требуется выбрать правильный перевод.
- Неверные ответы визуально подсвечиваются без перехода дальше.
- Прогресс обновляется корректно.
- Smoke e2e для прод-подпапки проходит.

## Rollout Plan

1. Реализовать `gerund-only` + `both-forms` + `mix`.
2. Выложить на прод.
3. Получить UX-апрув.
4. Раскатить тот же паттерн на `infinitive-only` и `bare-infinitive`.
